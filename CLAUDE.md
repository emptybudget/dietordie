# 지속가능 다이어트 기록 앱

계정 없이 누구나 URL만 열어 바로 쓰는 다이어트 로그. 음식·운동·수면·컨디션을 최소 마찰로 기록하고,
기간별 기록을 프롬프트로 묶어 외부 AI에 복붙 → 받은 요약을 다시 붙여넣어 보관한다.
모든 UI 판단 기준은 두 가지: **처음 온 사람이 설명 없이 쓸 수 있는가**, **매일 10초 안에 기록되는가**.
상세 계획·스키마·공개 기준선·빌드 순서는 PLAN.md 참조.

**본질: AI에게 내 데이터를 먹이기 쉽게 만드는 메모장.**
루프는 하나다 — ① 여기에 적는다 → ② 기록을 복사한다 → ③ 쓰는 AI에 먹인다 → ④ 받은 결과를 다시 여기에 저장한다.
기능 추가 판단 기준: 이 루프의 어느 단계를 더 매끄럽게 하는가. 어느 단계도 돕지 않으면 만들지 않는다.

## 행동 원칙 (모든 작업에 적용)

1. **Think before coding** — 구현 전에 가정을 명시한다. 해석이 갈리면 고르지 말고 묻는다.
   더 단순한 방법이 있으면 먼저 제안한다. 불명확하면 멈추고 무엇이 불명확한지 말한다.
2. **Simplicity first** — 요청한 것만 만든다. 단일 사용 코드에 추상화 금지.
   요청 없는 유연성·설정 옵션 금지. 불가능한 시나리오의 에러 처리 금지.
   200줄이 50줄로 될 수 있으면 다시 쓴다. "시니어가 보면 과하다고 할까?" — 그렇다면 단순화.
3. **Surgical changes** — 요청과 무관한 코드·주석·포맷은 건드리지 않는다.
   기존 스타일을 따른다. 내 변경이 만든 고아(import·변수·함수)만 정리하고,
   원래 있던 죽은 코드는 언급만 하고 지우지 않는다.
   모든 변경 줄은 사용자의 요청으로 직접 추적 가능해야 한다.
4. **Goal-driven** — 작업마다 검증 기준을 먼저 정의하고 통과할 때까지 반복한다.
   각 단계의 검증 항목은 PLAN.md의 빌드 순서 표를 따른다.

## 만들지 않는 것 (명시적 요청 전까지 금지)

- 칼로리/음식 DB, 영양소 계산
- 로그인/회원가입 (Phase 2 동기화도 계정 없는 토큰 방식, 공개 배포에는 미노출)
- 차트/그래프 라이브러리
- CSS/JS 프레임워크, 빌드 도구, npm 의존성 (Phase 1은 dependency 0개)
- 사진 업로드, 푸시 알림
- 외부 AI API 직접 호출 (프롬프트 복사 → 붙여넣기 방식 유지)
- 온보딩 모달·코치마크·튜토리얼 (빈 화면의 안내 문구로 해결한다)
- 다크모드, 다국어, 장식성 애니메이션

## 기술 제약

- Phase 1: 바닐라 HTML/CSS/JS, 빌드 스텝 없음, localStorage 저장, Vercel 정적 배포
- localStorage 접근은 반드시 storage.js 모듈 경유 (Phase 2 동기화 교체 지점)
- 파일 최소 유지: index.html, style.css, app.js, storage.js (+ 5단계에서 manifest.json, sw.js)
- 모바일 우선 UI (기록 입력은 대부분 폰에서 일어난다), 터치 타겟 44px 이상
- UI 문구는 전부 한국어, 기술 용어 노출 금지 ("localStorage" ✗ → "이 기기에 저장" ✓)

## 데이터 규칙

- 날짜 키는 YYYY-MM-DD, 사용자 로컬 시간 기준
- 수면은 **기상일**에 귀속. start > end이면 start는 전날 밤으로 해석
- 모든 day 객체에 updatedAt(ISO) 포함 (Phase 2 last-write-wins용)
- 스키마 변경 시 meta.schemaVersion 증가 + 마이그레이션 함수 작성 (opus 세션에서)

## 완료 기준

- 단계 완료 = PLAN.md 해당 단계의 검증 통과 + 새로고침 후 데이터 유지 확인
- 최종 완료(공개 가능) = PLAN.md **'공개 기준선'** 체크리스트 전 항목 통과
